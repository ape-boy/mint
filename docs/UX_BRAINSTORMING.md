# MintPortal UX 브레인스토밍 문서

## 1. 서비스 목적 및 도메인 컨텍스트

### 1.1 서비스 목적
**MintPortal**은 SSD 펌웨어 개발 조직을 위한 CI/CD 빌드 및 릴리즈 관리 시스템입니다.

### 1.2 핵심 도메인 모델

```
Task Group (제품 라인)       예: BM1743, PM9E1, PM9A3
    └── Project (세부 구성)   예: BM1743_LENOVO_1TB, BM1743_DELL_2TB
        └── Layer (빌드 유형)  예: Release, HIL, FIL, FTL
            └── Build (빌드)   파이프라인 실행 단위
```

### 1.3 핵심 사용 시나리오

| 시나리오 | 빈도 | 관련 사용자 |
|---------|------|-----------|
| 빌드 실행 및 모니터링 | 매일 | 개발자 |
| 빌드 결과 확인 | 매일 | 개발자, QA |
| 릴리즈(TR) 요청 | 주 1-2회 | 개발자, TL |
| 품질 지표 확인 | 주 1-2회 | TL, PM |
| 프로젝트 설정 변경 | 드물게 | TL, Admin |
| 다수 프로젝트 일괄 작업 | 이벤트성 | TL, PM |

## 2. 사용자 페르소나 분석

### 2.1 개발자 (Developer)

**목표:**
- 담당 프로젝트의 빌드를 빠르게 실행
- 빌드 결과/로그 확인
- 품질 지표(Coverity, SAM) 통과 여부 확인

**Pain Points:**
- 프로젝트 찾는 데 시간 소요
- 빌드 설정 변경이 필요할 때 어디서 하는지 모름
- 다른 Layer로 전환하려면 여러 클릭 필요

**Key Tasks:**
1. 내 프로젝트 빌드 실행
2. 빌드 결과/로그 확인
3. Pipeline 진행 상황 모니터링
4. Binary 다운로드

### 2.2 테크 리드 (TL - Tech Lead)

**목표:**
- 팀 프로젝트들의 빌드 상태 모니터링
- 릴리즈 승인/TR 처리
- 품질 지표 관리

**Pain Points:**
- 다수 프로젝트를 한눈에 파악하기 어려움
- 릴리즈 조건 미충족 시 결재 flow가 복잡
- 팀 전체 빌드 현황 대시보드 부재

**Key Tasks:**
1. 팀 프로젝트 빌드 상태 일괄 확인
2. 릴리즈 승인
3. 품질 지표 리포트 확인
4. 프로젝트 설정 관리

### 2.3 프로젝트 매니저 (PM)

**목표:**
- 전사 프로젝트 현황 파악
- 마일스톤 진행 상황 확인
- 품질/KPI 리포트

**Pain Points:**
- 프로젝트 그룹(제품군)별 필터링이 어려움
- 전사 품질 지표 대시보드 부재
- 마일스톤 대비 진행률 파악 어려움

**Key Tasks:**
1. 전사 프로젝트 현황 조회
2. 마일스톤 진행 상황 확인
3. KPI 리포트 확인

### 2.4 관리자 (Admin)

**목표:**
- 시스템 설정 관리
- 사용자/권한 관리
- 프로젝트/그룹 생성

**Pain Points:**
- 프로젝트 생성 flow가 복잡
- 권한 설정이 직관적이지 않음

## 3. 현재 정보 구조 분석

### 3.1 현재 메뉴 구조

```
MintPortal
├─ PROJECT
│   ├─ Project List ──────────→ / (메인)
│   └─ [프로젝트 선택 시]
│       ├─ Summary & Quality
│       ├─ 빌드
│       └─ 기본 정보
├─ 대시보드 ──────────────────→ /dashboard
├─ 게시판
│   ├─ 공지사항
│   ├─ VOC
│   └─ Release Note
└─ 설정
```

### 3.2 문제점 식별

| 문제 | 설명 | 영향 |
|------|------|------|
| **Task Group 진입점 부재** | 제품군(BM1743 등)별 필터는 있지만, Group 레벨 뷰가 없음 | PM이 제품군별 현황 파악 어려움 |
| **개발자 Quick Access 부재** | 개발자가 담당 프로젝트에 빠르게 접근하는 방법 없음 | 매번 검색/필터 필요 |
| **일괄 작업 지원 부재** | 다수 프로젝트에 공통 패치 시 일괄 빌드/릴리즈 불가 | 20개 프로젝트 개별 작업 필요 |
| **Layer 전환 어려움** | 프로젝트 내 Layer 전환 시 새로 로드됨 | 불필요한 대기 시간 |

## 4. UX 개선 방향 검토

### 4.1 옵션 A: Task Group 중심 구조

```
MintPortal
├─ DASHBOARD ───────────────→ 전사 현황 대시보드
├─ TASK GROUPS
│   ├─ BM1743 ─────────────→ 제품군 대시보드
│   │   ├─ Overview (프로젝트 요약)
│   │   ├─ Projects (20개 프로젝트 목록)
│   │   └─ Batch Operations (일괄 작업)
│   ├─ PM9E1
│   └─ PM9A3
├─ MY PROJECTS ────────────→ 내 담당 프로젝트 (개발자용)
├─ BOARD
└─ SETTINGS
```

**장점:**
- 제품군별 관리가 명확
- PM에게 직관적인 구조
- 일괄 작업 기능 도입 용이

**단점:**
- 네비게이션 깊이 증가 (2→3 depth)
- Task Group이 많아지면 메뉴 복잡
- 개발자는 "내 프로젝트"로 가야 하는 추가 동선

### 4.2 옵션 B: Flat List + 강화된 필터 (현재 구조 확장)

```
MintPortal
├─ DASHBOARD ───────────────→ 전사 현황 + 내 프로젝트
├─ PROJECTS ────────────────→ 전체 프로젝트 목록
│   ├─ 필터: Task Group, OEM, Status, Assignee(me)
│   ├─ 뷰 모드: Table / Group by Task Group
│   └─ 일괄 선택 → Batch Build/Release
├─ BOARD
└─ SETTINGS
```

**장점:**
- 단순한 구조 유지
- 필터 조합으로 다양한 뷰 가능
- 개발자/PM 모두 같은 화면 사용 가능

**단점:**
- Task Group 레벨 대시보드 없음
- 필터 의존도 높음 (매번 설정 필요)
- 일괄 작업 UX가 직관적이지 않을 수 있음

### 4.3 옵션 C: 하이브리드 (권장)

```
MintPortal
├─ DASHBOARD ───────────────→ 전사 + 내 프로젝트 요약
│   ├─ My Recent Builds
│   ├─ My Assigned Projects
│   └─ Task Group 요약 카드
├─ PROJECTS
│   ├─ [All Projects View] ─→ 목록 + 필터
│   ├─ [Group by Task Group] → 그룹별 접이식 뷰
│   └─ [Project Detail] ───→ 상세 + 빌드
├─ BATCH CENTER ───────────→ 일괄 작업 전용
│   ├─ Select Projects
│   ├─ Batch Build
│   └─ Batch Release
├─ BOARD
└─ SETTINGS
```

**장점:**
- 대시보드에서 빠른 접근
- 필요한 사람에게 필요한 뷰 제공
- 일괄 작업 명시적 지원

**단점:**
- 메뉴 항목 증가
- "BATCH CENTER" 개념 학습 필요

## 5. 시나리오별 UX 분석

### 5.1 시나리오: 개발자 일일 업무

**현재 Flow:**
```
1. MintPortal 접속
2. Project List에서 내 프로젝트 검색
3. 프로젝트 클릭 → 상세 페이지
4. 빌드 탭 클릭
5. Layer 선택 (Release)
6. 빌드 실행 버튼 클릭
7. 결과 대기/확인
```
**클릭 수: 6회**

**개선 제안 Flow:**
```
1. MintPortal 접속 → Dashboard
2. "My Recent Projects" 섹션에서 바로 클릭
3. 빌드 탭에서 빌드 실행
```
**클릭 수: 3회**

### 5.2 시나리오: PM1743 공통 패치 일괄 릴리즈 (20개 프로젝트)

**현재 Flow (불가능 또는 매우 비효율):**
```
각 프로젝트마다:
1. Project List에서 프로젝트 검색
2. 프로젝트 상세 이동
3. 빌드 탭 이동
4. 빌드 실행
5. 결과 확인
6. TR 버튼 클릭 (조건 충족 시)
반복 20회
```
**예상 소요 시간: 1-2시간**

**개선 제안 Flow (Batch Center):**
```
1. Batch Center 이동
2. Task Group "BM1743" 선택 → 20개 프로젝트 표시
3. 전체 선택 (또는 조건부 선택)
4. "Batch Build" 실행 → 20개 동시 빌드
5. 결과 대시보드에서 일괄 모니터링
6. 조건 충족 프로젝트 선택 → "Batch TR" 실행
```
**예상 소요 시간: 10-15분**

### 5.3 시나리오: TL 팀 빌드 현황 확인

**현재 Flow:**
```
1. Project List 접속
2. 내 팀 프로젝트 필터 (TL 또는 P/J로)
3. 각 프로젝트 상태 확인 (목록에서)
4. 문제 있는 프로젝트 상세 진입
```

**개선 제안:**
```
1. Dashboard 접속
2. "My Team Projects" 섹션에서 요약 확인
3. 빌드 실패 프로젝트 바로 클릭
```

## 6. 핵심 UX 질문들

### Q1: Task Group은 어떤 레벨의 중요도를 갖는가?

**현실:**
- Task Group(제품군)은 조직 구조와 직결
- 같은 Task Group 내 프로젝트들은 공통 코드베이스를 공유하는 경우 많음
- 공통 패치, 일괄 릴리즈 상황이 자주 발생

**권장:**
- Task Group을 1급 시민(first-class citizen)으로 승격
- Group 레벨 대시보드/일괄 작업 지원

### Q2: "내 프로젝트"는 어떻게 정의되는가?

**옵션:**
1. **TL/PL 기준**: 내가 TL/PL인 프로젝트
2. **Member 기준**: 내가 멤버로 등록된 프로젝트
3. **최근 접근 기준**: 내가 최근 접근한 프로젝트
4. **사용자 설정**: 내가 즐겨찾기한 프로젝트

**권장:** 2+3+4 조합
- 기본값: Member로 등록된 프로젝트
- 추가: 최근 접근 프로젝트
- 옵션: 즐겨찾기 기능

### Q3: 일괄 작업의 범위는?

**가능한 일괄 작업:**
- Batch Build: 여러 프로젝트 동시 빌드 트리거
- Batch TR: 조건 충족 프로젝트 일괄 릴리즈
- Batch Config Update: 공통 설정 일괄 변경 (위험)

**권장:** Batch Build와 Batch TR만 초기 지원
- Config 변경은 위험도가 높아 개별 처리 권장

### Q4: Layer 선택의 기본값은?

**현재:** 첫 번째 Layer (Release) 또는 마지막 접근 Layer

**권장:**
- 개발자: 마지막 접근 Layer 기억 (localStorage)
- 기본값: Release Layer

## 8. Trade-off 분석

### 8.1 단순성 vs 기능성

| | 단순 (현재) | 기능 추가 |
|---|---|---|
| 학습 곡선 | 낮음 | 높아짐 |
| 일상 업무 효율 | 보통 | 높음 |
| 특수 시나리오 | 매우 비효율 | 효율적 |
| 유지보수 | 쉬움 | 복잡 |

**권장:** Phase 1-2까지 진행 후 사용자 피드백 수집

### 8.2 개발자 중심 vs PM 중심

| | 개발자 중심 | PM 중심 |
|---|---|---|
| 정보 구조 | Flat, 빠른 접근 | Hierarchical, 구조화 |
| 대시보드 | 내 작업 중심 | 전사 현황 중심 |
| 빌드 UX | 즉시 실행 | 승인 workflow |

**권장:** 대시보드 분리 또는 탭으로 역할별 뷰 제공

### 8.3 일괄 작업 위험도

| 작업 | 위험도 | 권장 |
|------|--------|------|
| Batch View | 없음 | 즉시 적용 |
| Batch Build | 낮음 | 확인 모달 후 실행 |
| Batch TR | 중간 | 2단계 확인 (선택 → 확인 → 실행) |
| Batch Config | 높음 | 지원하지 않음 권장 |

## 10. 결론

### 권장 방향

1. **Task Group을 1급 시민으로 승격**
   - Group 레벨 대시보드 및 필터 강화

2. **"My Projects" 개념 도입**
   - 개발자 중심 빠른 접근 지원

3. **Batch Center 신규 메뉴**
   - 일괄 작업 전용 공간 제공

4. **점진적 개선**
   - Phase 1 → 피드백 → Phase 2 순서

### 최종 권장 메뉴 구조

```
MintPortal
├─ DASHBOARD ───────────────→ 역할별 대시보드
│   ├─ [개발자] My Projects, Recent Builds
│   └─ [PM/TL] Team Overview, Quality Summary
├─ PROJECTS ────────────────→ 프로젝트 관리
│   ├─ All Projects (필터 + 뷰 전환)
│   ├─ By Task Group (그룹별 접이식)
│   └─ Project Detail (기존 유지)
├─ BATCH CENTER ───────────→ 일괄 작업 (Phase 2)
├─ BOARD ──────────────────→ 게시판
└─ SETTINGS ───────────────→ 설정
```

이 문서는 브레인스토밍 단계의 산출물로, 실제 구현 전 사용자 검증이 필요합니다.
